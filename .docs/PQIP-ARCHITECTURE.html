1. Manifesto

Core conviction: The GUI is a medium, not the end goal. Testing whether a page renders is smoke testing. Product quality testing means verifying that the user's intent — apply for a job, save a document, change a subscription — actually succeeds end-to-end: data persists, events fire, downstream systems update, error paths recover gracefully.

1.1 What This System Is

PQIP is a Product Quality Intelligence Platform — not a test framework, not a CI/CD gate, not an automation harness. It answers three questions that no existing tool answers together:

  1. "Is the product ready?" — composite readiness score across verification, validation, resilience, data integrity, and cross-product safety.
  2. "What will break next?" — predictive risk scoring from historical defect patterns, code change velocity, and flakiness analysis.
  3. "How do we stop making the same mistakes?" — closed-loop feedback from test results back into planning, linting, architecture, and sprint estimation.

1.2 Design Principles

PrincipleImplication
Test outcomes, not interfacesEvery journey must verify a state change, not just a page render. "Button exists" is not a test — "button click caused the expected mutation" is.
Knowledge must compoundEvery defect found must make the system permanently immune to that defect class. Defects → prevention rules → lint checks → architecture fixes.
LLM-assisted cold startWhen historical data is thin, LLM expertise fills the gap — generating specs, classifying defects, predicting risk. As data accumulates, deterministic heuristics replace LLM calls.
Fail-safe by defaultEvery subsystem fails gracefully. A broken test layer never crashes the pipeline. DB unreachable → skip writes, not crash. LLM unavailable → fall back to heuristics.
Cross-product by designLessons from one product propagate to all products. A bug found in JP is a warning for CC, not just a JP defect.
Business-readable outputEvery report must be understandable by a CEO. Technical details live in the detail layer; the summary is narrative-driven.

1.3 What This System Is NOT

  • Not a unit test framework. pytest handles that. PQIP tests product behaviour, not function correctness.
  • Not a CI/CD pipeline. GitHub Actions handles that. PQIP runs post-deploy (and pre-merge for affected journeys).
  • Not a monitoring system. PQIP is active testing (runs journeys), not passive monitoring (watches metrics). The UXObserver bridges both.
  • Not a performance testing tool. Load testing, concurrent user simulation, and latency benchmarking are out of scope for v1.

Layer 1 — Resilience & Security

Question answered: "If a user does something wrong, unexpected, or malicious — does the system reject it gracefully, hold its data integrity, and not expose sensitive information?"

1.1 Adversarial Input Testing

A new executor (api-test-runner) makes direct HTTP calls to product APIs using session tokens captured by the tree executor's trunk logins. It does not use the browser — it tests the API layer directly.

// Adversarial step in journey spec
{
  "type": "adversarial",
  "target": "profile_name_field",
  "payloads": [
    { "value": "<script>alert('xss')</script>", "expect": "rejected_or_sanitized" },
    { "value": "A".repeat(100000), "expect": "truncated_or_rejected" },
    { "value": "'; DROP TABLE users; --", "expect": "rejected" }
  ],
  "assert_no": ["stack_trace_in_response", "raw_sql_in_error", "500_status"]
}

1.2 Authorization Boundary Testing

Using the multi-account tree (4 trunks: admin, free, standard, pro), test cross-tier access:

{
  "type": "security",
  "action": "authorization_bypass",
  "as_user": "free_tier",
  "call": "POST /api/v1/pro-only-feature",
  "expect": { "status": 403, "body_contains": "upgrade" }
}

Also tests IDOR (user A accessing user B's resources by ID guessing) and session integrity (expired JWT, revoked tokens).

1.3 Dirty-State Testing

ScenarioSetupExpected
Stale sessionLogin, wait past session timeout, attempt actionRedirect to login, then return with data preserved
Back button after mutationSubmit form, browser back, browser forwardNo duplicate submission, page shows current state
Concurrent tabsEdit name in tab A, edit bio in tab B, save bothBoth changes persist, no overwrite
Interrupted uploadStart file upload, kill network mid-transferError with retry option, no corrupted partial stored

1.4 Error Path Verification

For every journey that tests a happy path, auto-generate an error-path counterpart: submit with missing required fields, trigger network timeout, exceed rate limit. Verify the error UX is correct — message is user-readable, no technical jargon, includes an action ("Try again" / "Contact support").

LLM Bootstrap: At cold start, the LLM generates adversarial payloads per field type (text → XSS/SQL injection, email → format violations, file → type/size abuse). As the system runs, successful payloads are promoted to a deterministic payload library. After 50+ runs, the LLM is only invoked for novel field types.

Layer 2 — Behavioral Verification (Current)

Question answered: "Does the feature work correctly from the user's perspective — not just render, but actually deliver the intended outcome?"

2.1 Current Architecture (Implemented)

The tree executor (cc-test-runner) runs 26-node dependency trees via Claude Code subprocesses with Playwright MCP. Trunks log in, branches navigate, leaves verify features. Browser state is injected via --storage-state and --save-session flags.

2.2 Data Persistence Verification (Enhancement)

After every mutation step (form submit, file upload, config change), the runner:

  1. Navigates away from the current page (to /dashboard or another route)
  2. Navigates back to the mutation page
  3. Re-asserts the mutated data is still present
// Persistence assertion in journey spec
{
  "persistence": {
    "after_step": "save-profile",
    "navigate_away": "/dashboard",
    "navigate_back": "/profile",
    "assert": "headline field contains '{{expectedHeadline}}'"
  }
}

This catches: optimistic UI without backend save, session-only state, stale cache, race between save and subsequent read.

2.3 Narrative-Driven Scenarios (Enhancement)

Beyond individual feature tests, full user-narrative scenarios that create their own data and flow through features in the order a real user would encounter them:

2.5 Statistical Confidence (Enhancement)

Run each critical journey N times (default: 5) instead of once. Compute Wilson confidence interval:

Journey: jp-verify-job-apply
  Runs: 5 | Passes: 4 | Fails: 1 (network timeout attempt 3)
  Confidence: 80% ± 17% (Wilson)
  Classification: FLAKY — investigate timeout

Journey: jp-verify-tier-pro
  Runs: 5 | Passes: 5 | Fails: 0
  Confidence: >93% (lower bound Wilson, 0 failures in 5 trials)
  Classification: STABLE

Layer 3 — Integration & Flow Testing

Question answered: "Do the modules, services, and systems that compose the product actually work together — not just individually, but as a chain?"

3.1 QA Probe Endpoint

Each product backend exposes a lightweight probe endpoint (guarded by PHRONEX_QA_PROBE_ENABLED=true — stripped from production):

GET /api/v1/internal/qa-probe?request_id={uuid}

Response:
{
  "request_id": "abc-123",
  "events_fired": ["ApplicationCreated", "NotificationQueued"],
  "services_called": ["phronex-auth:verify_token", "notification-router:send"],

    { "table": "job_applications", "operation": "INSERT", "count": 1 },
    { "table": "notifications", "operation": "INSERT", "count": 1 }

The test runner captures the X-Request-Id header from the browser response, then queries the probe to verify the full service chain fired.

3.3 Cross-Product Contract Testing

Products communicate via shared contracts (JWT claims, API shapes, event schemas). Changes to a contract must be tested against all consumers:

# Contract definition: contracts/phronex_auth_jwt.yaml
name: phronex_auth_jwt
producer: phronex-auth
consumers: [jp, cc, portal, praxis, comc]
schema:
  required_claims: [sub, iss, exp, product_slug, tier]
  iss_must_be: "auth.phronex.com"
test: "Token generated by producer accepted by every consumer"

When any product's auth middleware changes, the contract test verifies all consumers still accept the token format.

3.5 Data Invariant Testing

Business rules that must hold true at all times, checked after every mutation step:

# Data invariants: invariants/jp_business_rules.yaml
invariants:
  - name: salary_range_valid
    table: job_postings
    rule: "salary_min <= salary_max OR salary_max IS NULL"
    severity: CRITICAL

  - name: tier_consistency
    tables: [subscriptions, razorpay_payments]
    rule: "subscription.tier matches latest successful payment plan_id"
    severity: CRITICAL

  - name: application_count_matches
    tables: [job_applications, dashboard_stats]
    rule: "COUNT(applications WHERE user_id=X) = dashboard_stats.applied_count"
    severity: HIGH
LLM Bootstrap: At cold start, the LLM reads the product's DB schema (via Alembic models or \d table_name) and generates candidate invariants. The operator reviews and approves invariant candidates. As invariants are validated across runs, confidence scores increase and false-positive invariants are pruned automatically.

4.2 Auto-Generated Journey Specs

When the coverage analyzer finds an UNTESTED feature, the system generates a draft journey spec from the USER-SPEC description + codebase analysis:

  1. Parse the USER-SPEC feature description
  2. Find the backend route handler (from route scan)
  3. Find the portal component (from component scan)
  4. Generate a multi-step journey spec with: navigation, interaction, assertion, persistence check
  5. Submit as a queued_for_approval proposal — operator reviews before it runs
LLM Bootstrap: Auto-spec generation relies heavily on LLM understanding of the USER-SPEC prose and the route handler's purpose. Early specs will be rough — the operator refines them. Each refinement is learned: the LLM sees what the operator changed and adjusts future generations. After 30+ approved specs, the LLM generates specs that require minimal edits. After 100+, the LLM may be replaced with template-based generation for common patterns.

4.3 Journey Depth Scoring

Every journey spec is automatically classified by depth:

DepthDefinitionJourneyHawk Policy
SMOKEOnly checks page loads (navigate, verify element exists)REJECTED — belongs in CI, not here
SURFACEClicks buttons, fills forms, checks immediate responseFLAGGED — should be deepened with persistence/integration assertions

5. Knowledge Compounding Model

5.1 The Exponential Decay Curve

Defect rate decreases exponentially as the knowledge base grows. This isn't aspirational — it's the mathematical consequence of permanently blocking each defect class.

MonthPrevention RulesLint ChecksData InvariantsDefect RateMonthly Hours/Product
1 (cold start)000100% baseline30h
3155870% (−30%)21h
640182540% (−60%)12h
1280+35+50+20% (−80%)6h

Across 6 products over 12 months:

  • Without compounding: 30 × 6 × 12 = 2,160 hours of defect work
  • With compounding: ~870 hours (60% cumulative reduction)
  • Savings: ~1,290 hours

5.2 What Makes This Exponential (Not Linear)

Traditional testing is linear: write more tests → find more bugs → fix them → same types recur. PQIP is exponential because each defect permanently removes a defect class:

  1. Test catches the bug (runtime detection)
  2. Pattern is learned (WikiStore knowledge capture)
  3. Pattern prevents recurrence (QA context injection into planning)
  4. Check blocks the code (lint rule, pre-commit gate)
  5. Architecture removes the class (structural elimination)

After stage 5, that entire class of bug cannot exist in the codebase. Not because developers memorised a rule, but because the system physically prevents the bad pattern from being committed.

5.3 Oracle and Spec Freshness

Static test oracles (TEST-ORACLES.html) and journey specs (jp-deep.json) decay as the product evolves. PQIP prevents oracle staleness through:

  • DocChain SHA256 tracking: when product .docs/ artefacts change, affected oracles are flagged for refresh
  • Auto-oracle generation: parse backend route handlers + frontend components to generate oracle rubrics that track the code, not a manual snapshot
  • Stale oracle detection: when a journey passes but the oracle hasn't been updated in 30+ days, flag as STALE_ORACLE — the pass may be against outdated criteria

6. LLM Bootstrap Strategy

Philosophy: The system starts LLM-heavy and becomes deterministic over time. LLM expertise fills gaps in historical data; as the data accumulates, heuristics replace LLM calls. Think of the LLM as the training wheels — essential at the start, removed when the system can ride on its own data.

6.1 Where LLM Assists at Cold Start

RCA Engine Classifies defects when deterministic heuristics don't match (currently 10 heuristic patterns) Each LLM classification is reviewed; confirmed matches become new heuristic rules Heuristic hit rate > 90% (LLM fallback fires <10% of the time)
Journey Spec Generation Generates draft journey specs from USER-SPEC.html prose + codebase analysis Operator edits are tracked; common patterns become template-based generation 30+ approved specs with <20% operator edit rate
Adversarial Payload Generation Generates adversarial payloads per field type (XSS, SQL injection, format violations) Successful payloads promoted to deterministic payload library Payload library covers >95% of field types encountered
Data Invariant Generation Reads DB schema and generates candidate business rule invariants Validated invariants become permanent; false-positive invariants pruned Invariant library covers all tables; new tables use templates
Prevention Rule Synthesis Summarises a defect + its fix into a generalised prevention rule Rules with high confidence become CODING-PATTERNS.md entries WikiStore confidence > 0.80 for the domain → rule is permanent
Business Readiness Report Translates technical test results into CEO-readable narrative summaries Common summary patterns become template-based; LLM handles novel situations Never fully retired — narrative quality benefits from LLM even with templates
Predictive Risk Scoring Estimates bug probability from commit diffs when historical data is sparse Statistical model trained on actual defect→commit correlations replaces LLM 50+ data points per code area → statistical model reliable

6.2 Cost Management

LLM calls are gated by the existing phronex_common.llm factory with capability-based routing and budget caps:

  • RCA fallback: uses Claude Haiku (cheapest) — classification only, no generation
  • Spec generation: uses Claude Sonnet — needs understanding of USER-SPEC prose
  • Report generation: uses Claude Sonnet — narrative quality matters
  • Budget cap: max $5/run across all LLM calls. If exceeded, skip LLM-dependent components (deterministic pipeline continues)
  • Trending toward zero: as deterministic alternatives mature, monthly LLM spend drops from ~$50 (month 1) to ~$5 (month 12) to ~$2 (month 18, reports only)

6.3 LLM-Generated Content Review Flow

All LLM-generated artefacts go through operator review before becoming permanent:

LLM generates → qa_proposed_heuristics (status: queued_for_approval)
                → Operator reviews in portal QA tab
                → Approved: applied + confidence += 0.10
                → Rejected: archived + LLM receives rejection feedback
                → After 5 approvals with <10% rejection rate:
                  auto-approve for this pattern class (operator still notified)

7. Peripheral Systems

PQIP doesn't operate in isolation. Several peripheral systems must evolve to support the full architecture:

7.1 DocChain (Enhanced)

Current: generates and reconciles .docs/ HTML artefacts (USER-SPEC.html, ARCHITECTURE.html, ROADMAP-NARRATIVE.html). Stores SHA256 snapshots for change detection.

Enhanced for PQIP:

  • Feature extraction: DocChain parses USER-SPEC.html into a structured feature inventory (Section → Feature → Capability → Acceptance Criteria). This is the input for Layer 4's coverage analyzer.
  • Change-to-journey mapping: when a USER-SPEC section changes, DocChain identifies which journey specs are affected and flags them for re-validation via the run filter (Reason B).
  • Oracle generation: DocChain generates TEST-ORACLES.html from the feature inventory + codebase analysis, ensuring oracle rubrics track the current spec rather than a stale manual snapshot.
  • Precision upgrade: current DocChain uses section-level SHA256 to detect changes. Enhanced DocChain uses paragraph-level diffing to pinpoint exactly which capability changed, enabling surgical journey re-runs instead of full-suite re-validation.

7.2 QA Probe Endpoints (New — per product)

Each Phronex backend needs a QA probe endpoint for Layer 3 integration testing. Specification:

AspectSpecification

GuardPHRONEX_QA_PROBE_ENABLED=true env var. Endpoint returns 404 when disabled. Must NEVER be enabled in production.

7.4 GSD Integration (Enhanced)

Current qa_context_hook.py provides a flat context block. Enhanced integration:

  • Risk-aware phase planning: hook computes risk score per affected file based on historical defect density
  • Required test coverage: hook specifies which journey specs must exist/pass before the phase can be marked complete
  • Sprint estimation adjustment: hook provides a multiplier based on code area rework rates
  • Anti-pattern injection: hook includes specific code patterns to avoid with evidence from past defects

7.5 Portal QA Dashboard (Enhanced)

The existing portal QA tab (/admin/qa) displays defect counts and run history. Enhanced to show:

  • Composite readiness score per product (all 10 dimensions)
  • Feature coverage heat map (USER-SPEC → journey → pass/fail)
  • Proposal approval queue (operator reviews LLM-generated specs, invariants, prevention rules)
  • Knowledge compounding dashboard (prevention rules created over time, defect rate trend)
  • Business readiness report (CEO view — demo-safe flows, investor readiness)

7.6 Predictive Risk Engine (New)

Analyses git diffs pre-merge to predict which journeys are likely to break:

  1. Parse the commit diff → identify changed files, routes, components
  2. Map changed files to journey specs that exercise them (via route↔journey index)
  3. Query defect history for the changed code area (defect density, flakiness score)
  4. Compute risk score: P(break) = base_rate × area_multiplier × change_size_factor
  5. Output: ranked list of journeys most likely to fail, with confidence intervals
LLM Bootstrap: With < 30 data points, the statistical model is unreliable. LLM reads the diff and makes a judgment call ("this changes the subscription webhook handler — historically fragile"). After 50+ merges with associated journey outcomes, the statistical model takes over.

8. Journey Spec Format v2

The journey spec format extends from the current flat step list to support all four test layers in a single file:

{
  "id": "jp-verify-profile-update",
  "role": "verify",
  "dependsOn": "jp-branch-profile",
  "params": { "expectedHeadline": "QA Test Headline" },

  // Layer 2 — Behavioral verification (current)
  "steps": [
    { "id": "1", "description": "Edit headline to '{{expectedHeadline}}'" },
    { "id": "2", "description": "Click Save button" },
    { "id": "3", "description": "Verify success toast appears" }
  ],

  // Layer 2 — Persistence check (new)
  "persistence": {
    "after_step": "2",
    "navigate_away": "/dashboard",
    "navigate_back": "/profile",
    "assert": "headline field contains '{{expectedHeadline}}'"
  },

  // Layer 2 — UX quality assertions (new)
  "ux": {
    "response_time_max_ms": 2000,
    "feedback_visible_within_ms": 500,
    "error_message_must_not_contain": ["SQL", "constraint", "500"]
  },

  // Layer 3 — Integration verification (new)
  "integration": {
    "after_step": "2",
    "probe": {
      "expect": {
        "db_writes": [
          { "table": "user_profiles", "column": "headline",
            "value": "{{expectedHeadline}}" }
        ]
      }
    }
  },

  // Layer 3 — Data invariants (new)
  "invariants": ["profile_completeness_score_updated"],

  // Layer 1 — Adversarial testing (new)
  "adversarial": [
    {
      "target": "headline_field",
      "payloads": [
        { "value": "<script>alert(1)</script>", "expect": "sanitized" },
        { "value": "x".repeat(10000), "expect": "truncated_or_rejected" }
      ]
    }
  ],

  // Layer 1 — Security (new)
  "security": {
    "authorization": {
      "as_different_user": true,
      "action": "GET /api/v1/users/{{userId}}/profile",
      "expect_status": 403
    }
  }
}

Backward compatibility: all new fields are optional. Existing specs with only steps continue to work — they just test at Layer 2 only. The depth scorer flags them as SURFACE.

10. Composite Readiness Scoring

10.1 Ten Dimensions

Each product receives a score across 10 dimensions. The composite score is a weighted average:

DimensionWeightSource
Verification (does it work?)20%qa_journey_verdicts pass rate
Validation (right thing built?)15%qa_feature_coverage coverage %
Data integrity15%qa_data_invariants pass rate
Resilience & security10%qa_adversarial_results pass rate
Cross-product safety10%qa_contract_tests pass rate
User experience quality10%UX assertion pass rate + response time compliance
Predictive intelligence5%Predicted risk vs actual outcome correlation
Confidence (statistical)5%Average Wilson confidence lower bound
Pre-deployment gating5%% of PRs that passed gate before merge
Business readability5%Report freshness + narrative coverage
