3. Resource Accumulation Layer NEW

Why this matters: Every JourneyHawk run depends on pre-existing resources — test accounts that can log in, seed data that populates UI components, dummy files that can be uploaded, admin credentials that grant access. Today these are scattered across .qa.env, hardcoded in spec files, and a single test PDF. When any resource goes stale (password changed, account deleted, file corrupted), entire run trees fail silently. The Resource Accumulation Layer makes test resources a first-class, inventoried, lifecycle-managed concern.

-- Migration: 0015_qa_test_resources.py
CREATE TABLE qa_test_resources (
    resource_id     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    category        TEXT NOT NULL,   -- 'account' | 'credential' | 'document' | 'seed_data' | 'config' | 'infrastructure'
    resource_key    TEXT NOT NULL,   -- unique within category, e.g. 'jp_free_tier_account'
    product_slug    TEXT NOT NULL,   -- which product this serves ('*' = all products)
    display_name    TEXT NOT NULL,   -- human-readable: 'JP Free Tier Test Account'

    -- Provider info
    provider_type   TEXT NOT NULL,   -- 'env_var' | 'file' | 'api_provision' | 'db_seed' | 'health_check'
    provider_config JSONB NOT NULL DEFAULT '{}',
    -- For env_var: {"var_name": "QA_JP_FREE_PASSWORD", "fallback_var": "PHRONEX_PORTAL_TEST_PASSWORD"}
    -- For file:    {"path": "fixtures/test-resume.pdf", "sha256": "abc...", "min_size_bytes": 100}
    -- For api_provision: {"endpoint": "/api/admin/test-setup/create-account", "payload": {...}, "cleanup_endpoint": "..."}
    -- For db_seed: {"table": "user_profiles", "query": "SELECT 1 FROM accounts WHERE email = %s", "params": ["qa-jp-free@phronex.com"]}
    -- For health_check: {"url": "https://jobc.phronex.com/health", "expected_status": 200}

    -- State tracking
    status          TEXT NOT NULL DEFAULT 'unknown',  -- 'available' | 'expired' | 'missing' | 'unknown' | 'provisioning'
    last_verified   TIMESTAMPTZ,
    last_used       TIMESTAMPTZ,
    expires_at      TIMESTAMPTZ,      -- NULL = never expires
    verification_error TEXT,           -- last error message if status != 'available'

    -- Lifecycle
    auto_provision  BOOLEAN NOT NULL DEFAULT FALSE,  -- can the system create this automatically?
    auto_cleanup    BOOLEAN NOT NULL DEFAULT FALSE,  -- should the system clean this up after each run?
    required_by     TEXT[] NOT NULL DEFAULT '{}',     -- journey IDs that need this resource

    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    UNIQUE(category, resource_key)
);

CREATE INDEX idx_qa_test_resources_product ON qa_test_resources(product_slug);

3.3 Resource Provider Pattern

# phronex_common/testing/resources/providers.py

class ResourceProvider(Protocol):
    """Verifies and optionally provisions a test resource."""
    def verify(self, config: dict) -> VerifyResult: ...
    def provision(self, config: dict) -> ProvisionResult: ...
    def cleanup(self, config: dict) -> None: ...

@dataclass(frozen=True)
class VerifyResult:
    available: bool
    error: str | None = None
    expires_at: datetime | None = None

# Concrete providers:
class EnvVarProvider(ResourceProvider):
    """Checks environment variable is set and non-empty."""

class FileProvider(ResourceProvider):
    """Checks file exists, meets min size, SHA256 matches."""

class APIProvisionProvider(ResourceProvider):
    """Calls product API to create/verify test resources.
    verify: GET /api/admin/test-setup/verify?resource_key=...
    provision: POST /api/admin/test-setup/create
    cleanup: POST /api/admin/test-cleanup/..."""

class DBSeedProvider(ResourceProvider):
    """Runs a SELECT query to verify seed data exists."""

class HealthCheckProvider(ResourceProvider):
    """TCP/HTTP health check against a URL."""

-- Migration: 0016_qa_feature_coverage.py
CREATE TABLE qa_feature_coverage (
    coverage_id     SERIAL PRIMARY KEY,
    product_slug    TEXT NOT NULL,
    feature_section TEXT NOT NULL,        -- USER-SPEC section heading
    feature_name    TEXT NOT NULL,        -- extracted capability name
    feature_hash    TEXT NOT NULL,        -- SHA256 of the feature description (staleness detection)

    -- Three verification levels
    code_exists     BOOLEAN,             -- backend route or portal component found?
    code_ref        TEXT,                -- e.g. "routes_profile.py:export_pdf" or "ProfileExportButton.tsx"
    spec_exists     BOOLEAN,             -- journey spec covers this feature?
    spec_ref        TEXT,                -- e.g. "jp-verify-profile-update"
    test_passes     BOOLEAN,             -- last journey verdict was PASS_ORACLE?
    last_tested     TIMESTAMPTZ,

    -- Metadata
    auto_detected   BOOLEAN DEFAULT TRUE,-- was this extracted automatically or hand-curated?
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    UNIQUE(product_slug, feature_name)
);

Implementation:

  1. Feature extraction — parse USER-SPEC.html using BeautifulSoup (already a phronex-common dependency). Walk <h2>/<h3> sections. Each bullet or paragraph describing a user capability becomes a feature row. At cold start, use LLM (Claude Haiku) to disambiguate "is this a discrete testable capability?" vs structural prose.
  2. Code existence check — for each feature, grep the product repo for matching route handlers (@router.post|get|put|delete) and portal components. Store first match as code_ref.
  3. Spec mapping — for each feature, search journey specs for ID or description overlap. Store matching journey ID as spec_ref.
  4. Verdict lookup — if spec_ref exists, query qa_journey_verdicts for latest verdict. Set test_passes accordingly.

Integration: Called from runner.py as new step in _phase82_seam(), after verdicts are written but before FeedbackConsolidator. Also callable standalone: python -m phronex_common.testing.coverage_analyzer --product jp

5.3 Task 85.3 — Composite Readiness Scorer (3h)

New file: phronex_common/testing/readiness_scorer.py

New table: qa_readiness_reports (migration 0017)

-- Migration: 0017_qa_readiness_reports.py
CREATE TABLE qa_readiness_reports (
    report_id       SERIAL PRIMARY KEY,
    product_slug    TEXT NOT NULL,
    run_id          TEXT,

    -- 10 dimension scores (0.0 to 1.0)
    verification_score      FLOAT,   -- qa_journey_verdicts pass rate
    validation_score        FLOAT,   -- qa_feature_coverage test_passes rate
    data_integrity_score    FLOAT,   -- qa_data_invariants pass rate (0.0 until Phase D)
    resilience_score        FLOAT,   -- qa_adversarial_results pass rate (0.0 until Phase E)
    cross_product_score     FLOAT,   -- qa_contract_tests pass rate (0.0 until Phase D)
    ux_quality_score        FLOAT,   -- UX assertion pass rate
    predictive_score        FLOAT,   -- risk prediction accuracy (0.0 until Phase G)
    confidence_score        FLOAT,   -- average Wilson lower bound (0.0 until Phase C)
    pre_deploy_score        FLOAT,   -- PR gate pass rate (0.0 until Phase F)
    readability_score       FLOAT,   -- report freshness (0 or 1)

    -- Composite
    composite_score FLOAT NOT NULL,  -- weighted average
    readiness_level TEXT NOT NULL,   -- SHIP | DEMO | DEVELOP | HOLD

    -- CEO-readable narrative (LLM-generated or template)
    narrative       TEXT,

    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

Scoring weights (per PQIP-ARCHITECTURE.html §10.1): Verification 20%, Validation 15%, Data integrity 15%, Resilience 10%, Cross-product 10%, UX 10%, Predictive 5%, Confidence 5%, Pre-deploy 5%, Readability 5%.

Dimensions that don't have data yet (Phases D/E/F/G not built) default to NULL, and the weight is redistributed proportionally across scored dimensions. This means Phase B's composite score is computed from verification + validation + UX only — three dimensions with a combined weight of 45%, normalized to 100%.

Integration: Called at the end of runner.py pipeline, after all other steps. Also callable standalone. The narrative field is generated by LLM (Claude Haiku) from the dimension scores — template-based for common patterns, LLM for novel combinations.

DoD — 85.3

6.2 Task 86.2 — Statistical Confidence & Flakiness Detection (4h)

New file: phronex_common/testing/confidence_scorer.py

New table: qa_confidence_scores (migration 0018)

-- Migration: 0018_qa_confidence_scores.py
CREATE TABLE qa_confidence_scores (
    score_id        SERIAL PRIMARY KEY,
    product_slug    TEXT NOT NULL,
    journey_id      TEXT NOT NULL,

    total_runs      INT NOT NULL DEFAULT 0,
    total_passes    INT NOT NULL DEFAULT 0,
    total_fails     INT NOT NULL DEFAULT 0,

    -- Wilson confidence interval (lower bound)
    confidence_lower FLOAT,         -- lower bound of Wilson interval
    confidence_upper FLOAT,         -- upper bound
    classification   TEXT,          -- 'STABLE' | 'FLAKY' | 'BROKEN' | 'INSUFFICIENT_DATA'

    -- Flakiness details
    last_flaky_run   TIMESTAMPTZ,   -- most recent run where result differed from previous
    flaky_streak     INT DEFAULT 0, -- consecutive alternating pass/fail runs

    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    UNIQUE(product_slug, journey_id)
);

Wilson interval formula: For p = passes/total, z = 1.96 (95% confidence):

lower = (p + z²/(2n) - z * sqrt((p*(1-p) + z²/(4n))/n)) / (1 + z²/n)

Classification rules:

  • STABLE: confidence_lower ≥ 0.80 AND no flaky_streak
  • FLAKY: 0.30 < confidence_lower < 0.80 OR flaky_streak ≥ 2
  • BROKEN: confidence_lower ≤ 0.30
  • INSUFFICIENT_DATA: total_runs < 3

Integration: Updated after every run by runner.py. Feeds into readiness_scorer (confidence dimension). Flaky journeys get priority in JourneyRecommender — they need more data points to converge.

DoD — 86.2

  • After 3 JP runs: SELECT journey_id, classification, confidence_lower FROM qa_confidence_scores WHERE product_slug='jp' ORDER BY confidence_lower LIMIT 5; returns rows with non-null classifications.
  • A journey that passed 5/5 times has classification=STABLE and confidence_lower > 0.80.
  • A journey that alternated pass/fail has classification=FLAKY.

8.8.2 Prevention Rules Management (sub-tab: “Prevention Rules”)

New sub-tab in QADashboardPanel (?qtab=prevention-rules). Surfaces the draft→active→retired lifecycle from Task 88.2.

  • Table view: All rules for selected product. Columns: rule_type badge, pattern_signature (monospace), source defect link, confidence bar, confirmed/FP counts, status badge (draft=amber, active=green, retired=gray).
  • Inline actions: “Activate” button (visible when confidence ≥0.75 + confirmed ≥2, status=draft). “Retire” button (visible when status=active). Both confirm via dialog before PATCH.
  • Stats header: Total rules, active count, retired count, total confirmed preventions.
  • Components: @phronex/ui DataTable, Badge, Button, ConfirmDialog.

8.8.3 Cross-Product Propagation Alerts (sub-tab: “Propagation”)

New sub-tab (?qtab=propagation). Shows defect patterns that were found in one product and detected in another’s codebase.

  • Alert cards: Source product badge → arrow → target product badge. Pattern signature in monospace. Match file + line number. Status badge (open=red, fixed=green, false_positive=gray).
  • Resolution actions: “Mark Fixed” and “False Positive” buttons per alert. Both PATCH to resolve endpoint.
  • Aggregate view: Product-pair matrix showing alert counts (e.g. JP→CC: 3 open, 1 fixed).
  • Components: @phronex/ui Card, Badge, Button, Alert.

8.8.4 CEO Readiness Report (enhancement to OverviewTab)

Enhances the existing OverviewTab (?qtab=overview) with executive-grade reporting. This is the view the CEO sees to understand product quality at a glance.

  • Readiness trend chart: Recharts AreaChart showing composite readiness score over time (from qa_readiness_reports). Shows threshold lines at 85% (SHIP), 70% (DEMO), 50% (DEVELOP).
  • Risk heat map: Treemap or grid of code areas sized by change frequency, coloured by defect rate (green <0.1, amber 0.1–0.3, red >0.3). Data from /api/v1/qa/risk-map.
  • Knowledge compounding gauge: Shows how many prevention rules are active vs total defects found — the “learning rate” of the QA system. Formula: active_rules / total_defects_found. Display as a progress ring with label “X% of defects have generated active prevention rules.”
  • Velocity metrics summary: Top 5 riskiest code areas by defect_rate. Sparkline of runs_to_stable trend per area.
  • Components: @phronex/ui Card, Badge. Recharts: AreaChart, Treemap (or custom grid), sparklines.
